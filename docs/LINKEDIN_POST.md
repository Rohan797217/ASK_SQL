## Post Option 1: Technical Deep-Dive (Recommended for engineering roles)

🚀 **I built a Text-to-SQL AI assistant that actually works in production — here's why most demos don't.**

Most "text-to-SQL" tutorials dump the entire database schema into an LLM prompt and pray. That works for a 3-table toy DB. In production? It fails spectacularly.

Here's what I built differently with **AskSQL**:

**❌ Problem 1: Context overflow**
Real databases have hundreds of tables. You can't fit them all in a prompt.
✅ **My fix:** Schema RAG — I embedded every table/column description into a Chroma vector store. When you ask a question, only the *relevant* tables get retrieved. Same idea as RAG for documents, applied to database schemas.

**❌ Problem 2: No safety**
LLMs will happily generate `DROP TABLE users` if you ask the right way.
✅ **My fix:** 5-layer guardrail system:
→ SQL parsing with sqlparse (SELECT-only enforcement)
→ 20+ blocked keyword detection
→ Multi-statement rejection
→ 12 SQL injection pattern checks
→ Auto LIMIT injection + read-only DB connection

**❌ Problem 3: Single point of failure**
One API goes down = your app breaks.
✅ **My fix:** Multi-provider LLM routing: Gemini → Groq → Ollama (local), with automatic fallback. Shows cost/reliability awareness — a real production concern.

**The result:** Ask a question in plain English → get SQL + real results from a live PostgreSQL database, with a polished React chat UI.
